# 需求分析示例

## 示例1：业务系统类需求

### 原始输入

用户说：  
“我要做一个线索池页面，运营可以查询、导入、新增线索，还能批量创建任务。”

### 参考输出

```markdown
## 一句话需求概述
新增一个线索池管理页面，支持运营人员对线索进行查询、维护、导入和任务创建。

## 需求目标
- 业务目标：提升线索统一管理和线下核查效率。
- 成功标准：运营可在一个页面完成线索检索、录入、导入和任务下发。

## 需求类型
- 类型：业务系统类
- 判断依据：需求核心是页面、列表、按钮、弹窗和任务操作。

## 适用角色
- 主要角色：运营人员
- 相关角色：调查员、任务处理人员
- 权限差异：待确认不同角色是否可编辑、删除、创建任务

## 核心场景
- 场景1：按筛选条件查询线索
- 场景2：手工新增线索
- 场景3：导入线索文件
- 场景4：勾选线索并批量创建任务
- 异常/边界场景：未勾选线索时创建任务、导入文件格式错误、附件超限

## 主流程
1. 入口：进入线索池页面
2. 用户动作：筛选、查看、导入、新增、勾选创建任务
3. 系统处理：查询数据、校验导入文件、保存线索、生成任务
4. 结果反馈：列表刷新、提示成功或失败原因

## 规则与约束
- 支持查询、重置、新增、导入、创建任务
- 导入文件格式、附件数量与大小限制需明确
- 批量创建任务前必须先选择线索
- 不同页签下的操作按钮可能不同

## 数据与依赖
- 关键字段：患者信息、机构信息、违规类型、结算日期、任务名称等
- 数据来源：线索池、任务模块、基础字典
- 外部依赖：文件上传能力、机构和用户数据源

## 范围界定
- 本次包含：线索池页面及基础操作闭环
- 本次不包含：待确认是否包含任务后续处理流程

## 风险与待确认问题
- 风险1：角色权限不明确会影响按钮展示和操作范围
- 待确认1：三个页签的状态定义和切换规则
- 待确认2：导入模板必填字段和失败反馈方式
```

## 示例2：规则逻辑类需求

### 原始输入

用户说：  
“要加一个违规判断规则，符合条件就提示违规，但有些情况要排除。”

### 参考输出

```markdown
## 一句话需求概述
新增一条违规判断规则，根据指定条件识别违规记录，并在例外场景下排除误判。

## 需求目标
- 业务目标：统一规则口径，减少人工判断成本。
- 成功标准：系统可稳定识别违规并输出明确提示。

## 需求类型
- 类型：规则逻辑类
- 判断依据：需求核心在于条件判断、触发逻辑、排除条件和结果输出。

## 适用角色
- 主要角色：审核人员
- 相关角色：规则维护人员
- 权限差异：待确认谁可查看命中结果、谁可维护规则

## 核心场景
- 场景1：命中规则并提示违规
- 场景2：满足排除条件，不提示违规
- 异常/边界场景：关键字段缺失、多个规则同时命中

## 主流程
1. 入口：系统处理待审核数据
2. 系统动作：读取输入字段并执行条件判断
3. 判断结果：命中则输出违规，不命中则继续
4. 结果反馈：记录规则说明、违规原因、命中依据

## 规则与约束
- 触发条件：待确认具体字段和判断阈值
- 判断逻辑：需要明确先后顺序和优先级
- 限制条件：关键字段缺失时如何处理
- 例外情况：满足排除条件时不判违规

## 数据与依赖
- 关键字段：患者、项目、时间、金额、机构等
- 数据来源：结算数据、规则配置、基础字典
- 外部依赖：规则引擎或规则配置模块

## 范围界定
- 本次包含：规则命中逻辑和提示输出
- 本次不包含：规则维护页面改造，除非用户额外提出

## 风险与待确认问题
- 风险1：没有明确优先级时，多个条件可能出现冲突
- 待确认1：命中后输出什么文案
- 待确认2：排除条件是否覆盖特殊业务场景
- 待确认3：字段缺失时是跳过、报错还是记为未命中
```

## 使用提示

- 当原始输入偏“页面功能”时，优先往页面、字段、交互、状态方向拆解。
- 当原始输入偏“规则判断”时，优先往条件、顺序、例外、结果方向拆解。
- 如果用户后续要写 PRD，可把此处产出的内容直接作为 PRD 初稿输入。
