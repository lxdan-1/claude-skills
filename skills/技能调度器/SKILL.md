# 技能调度器

## 使用目标

先做任务识别与编排，再调用最少且必要的下游 skills，避免每次都手动指定顺序。

## 调度原则

1. 先判断用户目标，不要一上来同时调用所有 skills。
2. 优先选择最短可行路径；只有当前置输入不足时，才补前置 skill。
3. 缺关键信息时先追问，不强行继续。
4. 每次开始执行前，用一句话说明本次将按什么顺序处理。
5. 每一步的输出，应尽量成为下一步的直接输入，减少重复改写。

## 任务分类

### 直接答疑

适用关键词：

- 是什么
- 怎么理解
- 帮我解释
- 差异是什么
- 为什么这样设计

默认调用：

- `问题答疑`

如果问题核心是规则口径，再切到：

- `规则逻辑速查`

### 规则梳理

适用关键词：

- 规则
- 条件判断
- 触发逻辑
- 命中口径
- 排除条件

默认调用：

- `规则逻辑速查`

如需沉淀成正式文档，再继续：

- `PRD写作`

### 模糊需求整理

适用关键词：

- 我有个想法
- 帮我整理需求
- 这个功能怎么做
- 帮我拆一下

默认调用：

- `需求分析`

如用户需要正式文档，再继续：

- `PRD写作`

如用户还要页面原型，再继续：

- `交互页面生成`

### 已有需求，输出 PRD

适用条件：

- 用户已有较完整需求分析
- 用户明确要求 PRD

默认调用：

- `PRD写作`

如果输入仍然零散，先补：

- `需求分析`

### 已有 PRD，输出页面原型

适用条件：

- 用户已有 PRD、页面说明或结构化页面需求
- 用户明确要 HTML 原型/交互页面

默认调用：

- `交互页面生成`

如果页面目标、角色、流程、规则仍不完整，先补：

- `需求分析` 或 `PRD写作`

### 评审与验收

适用关键词：

- 评审
- 检查
- 有没有问题
- 帮我 review
- 验收

默认调用：

- `评审检查`

如果评审对象只是零散描述，先补：

- `需求分析` 或 `PRD写作`

### 提示词与工作流优化

适用关键词：

- 提示词优化
- prompt 改写
- 工作流优化
- 提高稳定性

默认调用：

- `提示词优化`

如果目标是为一整条产品流程设计提示词链路，可联动：

- `技能调度器`

### 市场与竞品调研

适用关键词：

- 市场调研
- 竞品分析
- 行业扫描
- 方案对比

默认调用：

- `市场调研`

## 推荐链路

### 从想法到原型

`需求分析` -> `PRD写作` -> `交互页面生成` -> `评审检查`

### 从规则到规则文档

`规则逻辑速查` -> `PRD写作` -> `评审检查`

### 从问题到结论

`问题答疑`

如问题本质是规则说明：

`规则逻辑速查` -> `问题答疑`

### 从原始提示词到稳定工作流

`提示词优化` -> 必要时补充 `技能调度器`

## 输入不足时的处理

优先补这些高价值问题：

- 当前目标是答疑、整理需求、写 PRD、做原型，还是做评审？
- 你现在已有的是想法、会议纪要、PRD，还是现成页面/代码？
- 本次是否要直接产出文件？
- 是否已有上游材料路径可作为依据？

如果缺少关键前置材料，不要跳步。

## 输出约定

开始执行时，先用一句话声明本次路径，例如：

```markdown
本次按：需求分析 -> PRD写作 -> 交互页面生成。
```

若不继续下游步骤，也要明确停点，例如：

```markdown
本次只做：规则逻辑速查；暂不进入 PRD 写作。
```

## 质量检查

- 是否选了最少必要的 skills，而不是全量调用。
- 是否在信息不足时先追问。
- 是否按先后依赖顺序调用，而不是跳步。
- 是否让上一步输出可直接承接到下一步。
- 是否向用户清楚说明本次调用路径。
