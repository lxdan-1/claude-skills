# 评审检查示例

## 示例1：评审需求分析结果

### 输入摘要

```markdown
## 需求目标
- 提升线索统一管理效率

## 核心场景
- 查询线索
- 新增线索
- 导入线索

## 规则与约束
- 导入文件需要校验格式

## 范围界定
- 本次包含线索池页面
```

### 参考输出

```markdown
## 评审对象
- 文档名称：线索池需求分析
- 评审类型：需求分析

## 总体结论
当前材料已具备基础方向，但还不足以直接进入 PRD 写作，关键场景和规则约束仍需补充。

## 高优先级问题
1. 问题：核心场景只列出了查询、新增、导入，未说明是否包含批量创建任务、详情查看、编辑删除等核心闭环动作。
   影响：后续 PRD 范围可能被误缩，导致功能遗漏。
   建议：补齐主流程中的核心操作闭环，并明确每个场景的入口和结果。

2. 问题：规则与约束仅写“导入文件需要校验格式”，未说明允许格式、大小限制、失败反馈方式。
   影响：研发和测试无法统一实现与验收口径。
   建议：将规则展开为可执行表达，至少补齐格式、大小、必填字段和失败提示。

## 中低优先级问题
1. 问题：范围界定只写“本次包含线索池页面”，缺少“不包含”范围。
   影响：后续可能引发边界争议。
   建议：补充非本次范围，例如是否包含任务后续处理流程。

## 待确认项
- 是否存在角色权限差异
- 是否有异常场景和边界场景定义

## 评审建议
- 先补齐核心场景、规则细节和范围边界，再进入 PRD 写作
```

## 示例2：评审 PRD 文档

### 输入摘要

```markdown
## 三、用户与场景
- 主要角色：运营人员

## 四、方案说明
### 4.1 页面结构
- 筛选区
- 列表区

## 七、验收标准
1. 功能操作正常
2. 页面展示无问题
```

### 参考输出

```markdown
## 评审对象
- 文档名称：线索池页面 PRD
- 评审类型：PRD

## 总体结论
文档框架已形成，但交互细节和验收标准过于粗，当前不建议直接进入开发。

## 高优先级问题
1. 问题：方案说明只有页面结构，没有展开到业务用例、字段、按钮、状态和异常反馈。
   影响：研发无法据此实现，设计和测试也无法据此确认细节。
   建议：按业务用例补齐页面入口、筛选条件、功能按钮、表单字段和交互反馈。

2. 问题：验收标准写成“功能操作正常”“页面展示无问题”，缺少可验证条件。
   影响：测试无法据此设计用例，评审也无法判断交付完成度。
   建议：将验收标准拆成具体可验证项，例如导入校验、生效提示、未选线索时的报错等。

## 中低优先级问题
1. 问题：角色部分仅写运营人员，未说明调查员或其他关联角色。
   影响：权限和可见范围可能遗漏。
   建议：补充相关角色及权限差异说明。

## 待确认项
- 页面是否存在多页签和页签状态定义
- 是否支持批量操作及其限制条件

## 评审建议
- 优先补齐业务用例与验收标准，再组织复评
```

## 示例3：评审同目录下的 PRD + HTML 原型

### 输入摘要

```markdown
目录：05-项目输出/阶段产出/线索池PRD/

- 线索池页面PRD.md
- 线索池页面原型.html
- 线索新增抽屉原型.html

已知 PRD 要点：
- 支持查询、新增、导入、批量创建任务
- 不同页签下操作按钮不同
- 未勾选线索时，创建任务需给出明确提示

已知原型表现：
- 有列表页和新增抽屉
- 列表页支持页签切换
- 创建任务按钮始终可点击
```

### 参考输出

```markdown
## 评审对象
- 文档名称：线索池页面 PRD + 配套原型
- 评审类型：PRD+原型

## 配套材料
- PRD：`05-项目输出/阶段产出/线索池PRD/线索池页面PRD.md`
- HTML 原型：`05-项目输出/阶段产出/线索池PRD/线索池页面原型.html`
- HTML 原型：`05-项目输出/阶段产出/线索池PRD/线索新增抽屉原型.html`

## 总体结论
当前材料已形成成套交付，但 PRD 与原型之间仍存在关键不一致，暂不建议直接进入正式评审结论确认。

## 高优先级问题
1. 问题：PRD 明确要求“未勾选线索时，创建任务需给出明确提示”，但原型中“创建任务”按钮始终可点击，且没有禁用态或提示反馈。
   影响：评审结论可能误判为该交互已定义并可落地，后续实现与验收口径会偏差。
   建议：在原型中补齐禁用态或点击提示，并与 PRD 保持一致。

2. 问题：PRD 中定义了“导入”场景，但当前配套原型仅体现列表页和新增抽屉，未体现导入弹窗或导入流程。
   影响：成套交付不完整，评审时无法验证导入场景。
   建议：补充导入原型，或在 PRD 中明确该部分尚未出原型。

## 中低优先级问题
1. 问题：PRD 和原型文件命名尚可识别，但未形成统一前缀规范。
   影响：后续页面增多时，目录可读性会下降。
   建议：统一使用“线索池-页面名称-原型.html”之类命名方式。

## 待确认项
- 导入弹窗是否属于本轮原型交付范围
- 创建任务流程是否需要单独补原型页面

## 评审建议
- 先补齐 PRD 与原型的一致性，再组织成套评审复核
```

## 使用提示

- 对 `需求分析` 做评审时，重点抓“缺没缺”“清不清”。
- 对 `PRD写作` 做评审时，重点抓“能不能做”“能不能验”。
- 对 `PRD + HTML 原型` 做评审时，重点抓“是不是一套”“是否一一对应”“有没有互相打架”。
- 如果问题很多，优先输出高风险问题，不要先纠结措辞细节。
