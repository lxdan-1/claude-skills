# 评审检查

## 使用目标

对 `需求分析`、`PRD写作` 或配套 HTML 原型做系统性检查，优先发现遗漏项、歧义项、冲突项和高风险点，而不是只给笼统意见。

## 输入前提

优先评审以下材料之一：

- `需求分析` 的结构化输出
- `PRD写作` 产出的正式文档
- `交互页面生成` 产出的 HTML 原型文件
- 页面方案、流程说明、规则说明

如果同一目录下同时存在 PRD 和对应 HTML 原型，默认将它们视为同一组评审对象，优先做“成套评审”，而不是拆开单独评。

如果输入过于零散，先提醒补齐背景、目标、角色、流程、规则和范围，再开展正式评审。

## 适用场景

- 对 PRD、方案、流程、页面或规则进行评审。
- 对 PRD 与配套 HTML 原型进行一致性评审。
- 需要系统性检查遗漏项、歧义项和风险项。
- 需要形成可追踪的问题清单和评审结论。
- 需要在评审会上快速指出关键问题，而不是平均用力。

## 问题分级规则

默认分成三类：

### 高优先级问题

适用于：

- 目标不清，导致方案可能做错
- 范围不清，导致交付边界失控
- 主流程缺失，导致核心场景无法落地
- 规则冲突，导致实现或测试无法统一口径
- 验收标准缺失，导致无法判断是否完成

### 中低优先级问题

适用于：

- 文案不清
- 命名不统一
- 字段说明不完整
- 页面状态说明不足
- 示例和说明可进一步补强

### 待确认项

适用于：

- 当前材料无法确认，但会影响方案判断
- 需要业务方、设计、研发或测试进一步对齐

## 执行步骤

1. 明确评审对象，是需求分析结果、PRD、页面方案还是规则说明。
2. 如果同目录下存在 PRD 与 HTML 原型，先建立一一对应关系，再做成套评审。
3. 先看目标与范围：有没有目标不清、边界不清、范围漂移。
4. 再看角色与场景：有没有角色遗漏、场景不完整、权限未说明。
5. 再看流程与交互：有没有主路径缺失、异常路径遗漏、状态不闭环。
6. 再看规则与数据：有没有条件不全、口径冲突、字段缺失、依赖不明。
7. 再看实现与验收：有没有依赖风险、测试困难、验收标准不可执行。
8. 如果存在 HTML 原型，检查其与 PRD 是否一致，是否存在“文档写了但原型没体现”或“原型做了但 PRD 没定义”。
9. 输出问题清单，并按严重程度排序。

## 重点检查清单

### 目标与范围

- 需求目标是否明确。
- 是否能看出本次为什么做。
- 是否区分本次范围与非本次范围。
- 是否存在一句话写得很大、正文落得很散的问题。

### 角色与场景

- 主要角色是否明确。
- 是否存在角色权限差异但未说明。
- 是否覆盖正常、异常和边界场景。
- 是否遗漏关键上下游角色。

### 流程与交互

- 主流程是否完整。
- 页面入口、跳转、反馈是否说清。
- 是否覆盖默认态、空态、错误态、成功态。
- 是否存在用户动作和系统反馈断裂。

### 原型一致性

- PRD 中的页面入口、按钮、字段、状态，HTML 原型里是否有体现。
- HTML 原型中的交互行为，PRD 中是否有依据。
- 同目录下的 PRD 和 HTML 文件命名是否可对应。
- 是否存在 PRD 写了多页面，但原型只做了一半的情况。
- 原型中的模拟数据、状态和文案是否符合 PRD 设定。

### 规则与数据

- 规则是否写成可执行表达。
- 是否说明触发条件、判断顺序、例外情况。
- 关键字段是否完整，命名是否一致。
- 数据来源、接口依赖、外部依赖是否清楚。

### 实现与验收

- 是否存在技术或依赖风险。
- 验收标准是否能逐条验证。
- 测试是否能据此编写用例。
- 是否存在“写了方案但没法验收”的问题。

## 和前置 skill 的衔接规则

当评审对象来自 `需求分析` 时，重点检查：

- `需求目标` 是否真实回答了业务问题
- `核心场景` 是否覆盖正常、异常、边界场景
- `规则与约束` 是否足够支撑后续写 PRD
- `范围界定` 是否可用于控制项目边界
- `风险与待确认问题` 是否完整

当评审对象来自 `PRD写作` 时，重点检查：

- 背景与目标是否和需求分析一致
- 业务用例或规则说明是否能落地实现
- 交互、字段、状态、校验是否足够具体
- 验收标准是否可执行
- 待确认问题是否集中收口且未遗漏

当评审对象来自 `交互页面生成` 时，重点检查：

- HTML 原型是否和对应 PRD 同目录配套放置
- 页面入口、模块、字段、按钮、状态是否与 PRD 一致
- 关键交互是否真实可操作，而不只是静态摆设
- 模拟数据是否真实合理，并能支撑主要评审场景
- 原型是否覆盖主路径和至少一个异常或边界场景

当评审对象是“PRD + HTML 原型”成套材料时，重点检查：

- 两者的目标、范围、角色和核心场景是否一致
- PRD 中定义的页面和原型中的页面是否一一对应
- PRD 中定义的规则、字段、按钮、状态是否在原型中落地
- 原型中体现的交互和数据，是否超出了 PRD 已定义范围
- 同目录交付是否清晰，文件命名是否便于追踪

## 输出要求

默认使用以下结构：

```markdown
## 评审对象

## 配套材料

## 总体结论

## 高优先级问题

## 中低优先级问题

## 待确认项

## 评审建议
```

## 输出写法要求

- 每个问题尽量包含：问题、影响、建议动作。
- 能引用依据时要给出依据，不只下判断。
- 高优先级问题优先写业务影响和交付风险。
- 避免写成“感觉不太好”，要写成“为什么有问题”。
- 如果没有明显问题，也要说明残余风险和测试空白。

## 推荐模板

```markdown
## 评审对象
- 文档名称：
- 评审类型：需求分析 / PRD / HTML原型 / PRD+原型 / 页面方案 / 规则说明

## 配套材料
- PRD：
- HTML 原型：
- 其他材料：

## 总体结论
一句话说明当前材料是否可继续进入设计、开发或评审下一步。

## 高优先级问题
1. 问题：
   影响：
   建议：

## 中低优先级问题
1. 问题：
   影响：
   建议：

## 待确认项
- 待确认1：
- 待确认2：

## 评审建议
- 建议继续修改后复评 / 可进入下一阶段但需补充说明
```

## 质量检查

- 问题描述是否具体到可行动。
- 是否说明影响范围和可能后果。
- 是否优先指出关键风险，而不是平均用力。
- 是否区分“必须修改”和“建议优化”。
- 是否避免只给主观意见、不指出依据。
- 是否明确哪些属于缺陷，哪些属于待确认。
- 是否能帮助需求方、设计、研发、测试共同对齐。
- 如果存在 PRD 与 HTML 原型，是否检查了两者的一致性与同目录配套关系。
