# PRD写作

## 使用目标

把 `需求分析` 阶段输出的结构化信息，继续整理成可评审、可设计、可开发、可测试的 PRD 文档。

## 输入前提

优先接收以下输入：

- 一句话需求概述
- 需求目标
- 需求类型
- 适用角色
- 核心场景
- 主流程
- 规则与约束
- 数据与依赖
- 范围界定
- 风险与待确认问题

如果还没有这些内容，优先先走 `需求分析`，再进入本 skill。

## 交付物意识

默认不要把 PRD 视为单独文件，而要把它视为一组配套交付物中的核心文档。

对于页面型需求，默认至少考虑以下配套关系：

- `PRD.md`
- 页面原型 `.html`
- 必要时的评审记录或补充说明

其中 HTML 原型默认应与对应 PRD 放在同一目录，形成一套可追踪、可评审、可回看的需求产物。

## 先做判断

开始写之前，先判断文档主类型：

### 业务系统类 PRD

适用于：

- 页面
- 列表
- 表单
- 筛选
- 按钮
- 弹窗
- 跳转
- 详情

写作重点：

- 用户故事
- 页面结构
- 业务用例
- 字段说明
- 交互逻辑
- 状态变化
- 验收标准

### 规则逻辑类 PRD

适用于：

- 规则
- 判断
- 匹配
- 命中
- 违规
- 触发条件
- 排除条件

写作重点：

- 规则背景
- 适用范围
- 输入数据
- 判断逻辑
- 处理流程
- 输出结果
- 例外与优先级
- 验收标准

如果一个需求同时覆盖页面和规则，不要强行写成一块。优先按“页面方案”和“规则说明”分节，必要时拆成两份文档。

## 执行步骤

1. 先检查输入是否完整，缺背景、角色、流程、规则时先标待确认。
2. 根据需求类型选择文档主结构，不混用章节。
3. 先写业务目标和用户故事，确保文档先回答“为什么做”。
4. 把需求分析中的核心场景展开成业务用例或规则说明。
5. 把规则补成明确表达，尽量写成“触发条件 -> 处理逻辑 -> 输出结果”。
6. 页面类内容要补充字段、按钮、状态、异常反馈。
7. 对页面型需求，补充配套原型建议，确保后续 HTML 原型与 PRD 同目录交付。
8. 最后补验收标准、待确认问题和依赖风险。

## 业务系统类推荐结构

```markdown
# 【功能名称】

## 变更记录

## 1、用户故事

## 2、界面元素

## 3、业务需求说明

### 3.1、业务用例

### 3.2、业务规则

## 4、验收标准

## 5、附录

### 5.1、数据字典

### 5.2、状态流转

### 5.3、权限说明
```

## 规则逻辑类推荐结构

```markdown
# 【规则名称】

## 变更记录

## 1、规则背景与目标

## 2、适用范围

## 3、规则说明

### 3.1、规则定义

### 3.2、触发条件

### 3.3、处理逻辑

### 3.4、排除条件与优先级

### 3.5、输入输出说明

## 4、验收标准

## 5、附录
```

## 写作要求

- 先写目标，再写方案。
- 章节名称稳定，避免每次自由发挥。
- 避免“支持”“可灵活配置”这类空泛表达，改写成可验证规则。
- 页面类内容要说明入口、区域、按钮、字段、默认态、空态、异常态。
- 规则类内容要说明触发条件、判断顺序、排除条件、结果输出、数据来源。
- 无法确认的信息统一收口到“待确认问题”，不要散落全文。
- 页面型需求默认要考虑与 HTML 原型的配套关系，而不是只交付一份 PRD。

## 和需求分析的衔接规则

当输入来自 `需求分析` 时，按以下映射展开：

- `需求目标` -> PRD 的背景、目标、用户故事
- `适用角色` -> 用户角色、权限说明
- `核心场景` -> 业务用例或规则适用场景
- `主流程` -> 页面流程、处理流程、状态流转
- `规则与约束` -> 业务规则、规则说明、校验逻辑
- `数据与依赖` -> 数据字典、接口依赖、外部依赖
- `范围界定` -> 文档中的范围说明和非本次内容
- `风险与待确认问题` -> 待确认问题、风险提示、实现注意事项

## 和交互页面生成的衔接规则

当 PRD 用于后续生成 HTML 原型时，默认遵循以下配套原则：

- 如果已经确定 PRD 的输出目录，后续 HTML 原型默认与该 PRD 放在同一目录
- 如果是页面型需求，PRD 中应尽量写清页面入口、模块、字段、状态、按钮和规则，减少后续原型生成偏差
- 如果同一需求包含多个页面，可在 PRD 中提前约定页面名称，便于 HTML 文件命名
- 如果某些页面细节尚不确定，应在 PRD 中明确标记“待确认”，避免原型阶段误补

推荐目录关系：

```text
05-项目输出/阶段产出/某需求目录/
  某需求PRD.md
  某页面原型.html
  某弹窗原型.html
  评审记录.md
```

## 输出要求

默认输出以下结构：

```markdown
# 文档标题

## 一、背景与目标

## 二、范围说明

## 三、用户与场景

## 四、方案说明

## 五、规则与交互

## 六、数据与依赖

## 七、验收标准

## 八、待确认问题
```

如果用户明确要求按现有模板写，则优先遵循对应模板结构。

对于页面型需求，建议在输出完成后补一行交付说明，例如：

```markdown
## 交付物建议
- PRD 文件：与当前文档同目录保存
- 配套原型：后续生成的 HTML 原型与 PRD 同目录保存
```

## 验收标准写法

验收标准至少覆盖：

- 功能操作是否可完成
- 页面展示或结果输出是否正确
- 字段校验或规则判断是否正确
- 边界场景和异常场景是否可处理

## 质量检查

- 是否每个功能点都能对应到明确目标。
- 是否每条规则都具备条件、动作和结果。
- 是否减少模糊措辞，改用可验证表达。
- 是否区分“已知事实”和“待确认问题”。
- 是否能直接被研发、测试和评审人员理解。
- 是否补充了验收标准，而不是只写方案。
- 是否与 `需求分析` 结果前后一致。
- 对页面型需求，是否考虑了与配套 HTML 原型的目录和命名关系。
