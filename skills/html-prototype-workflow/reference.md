# 技术要求 → HTML 交互原型 · 七步工作流

> **这是什么？** 一套可复用的方法论，适用于**政务 / 医疗 / 监管类信息系统**的原型设计。从一份技术要求文档出发，高效生成符合甲方视角的高质量交互式 HTML 原型。
>
> **谁适合用？** 产品经理、售前顾问、系统分析师、投标方案制作者。

---

## 快速上手（3分钟版本）

```
我有一份政务信息系统的技术要求文档，请帮我生成一个完整的交互式HTML原型。

步骤：
1. 分析文档，识别系统用户（甲方角色）
2. 设计信息架构（参考：数据底座→核心业务→分析→处置→统计→系统管理）
3. 生成单文件HTML，包含：
   - 左侧 248px 深色侧边栏（用细分割线 menu-divider，不用 menu-section 分区标题）
   - 二级菜单用 data-page 属性绑定激活状态，展开高度 max-height ≥ 600px
   - 每个页面的筛选栏、KPI卡片（标签全部用 <div>）、数据表格
   - 表格超过4列时独占全宽；操作列加 white-space:nowrap
   - 用JS存储并渲染真实感数据（从技术要求中提取所有实体类型）
   - 关键操作有弹窗反馈
4. 只包含甲方用户会使用的功能，删除所有乙方运维指标和操作
5. 使用专业蓝色系政务配色（--primary:#1658d3, --bg:#eaeff8）

技术要求文档：
[粘贴你的技术要求内容]
```

## 核心原则：甲乙方视角分离（最重要，不可跳过）

```
甲方（医保局/监管部门）关心：
  ✅ 发现了多少违规线索
  ✅ 追回了多少资金
  ✅ 哪些医院风险最高
  ✅ 审核员今天要处理什么

乙方（供应商）才关心：
  ❌ AI模型准确率（内测数据）
  ❌ 实体抽取词条数量
  ❌ 系统解析成功率
  ❌ 项目实施进度
  ❌ 工单/培训/反馈管理
```

**识别乙方内容的口诀**：问自己"这个功能/数据，是甲方日常工作要用的，还是供应商技术人员在后台维护的？"——后者全部删除或替换。

├── 处置运营（线索→整改→闭环）

**模块命名原则：**

| 原则 | 说明 | 示例 |
|------|------|------|
| 面向业务结果 | 模块名反映用户"做什么" | "疑似违规分析" > "AI分析引擎" |
| 避免技术术语 | 局方不理解"NLP解析" | "审核结果概览" > "AI解析概览" |
| 按工作流排序 | 数据→审核→分析→处置→报告 | 即侧边栏顺序 |

## 易错点约束（生成前必读）

> 以下7条约束来自实际项目中真实踩过的坑，**生成前必须确认全部满足**。

### 生成前强制检查清单

```
□ 侧边栏宽度 ≥ 248px（中文子菜单项10字以上容易截断）
□ sub-item 颜色明度足够（推荐 #b8ccf0，禁用 rgba(255,255,255,.55)）
□ sub-menu open 状态 max-height ≥ 600px（6个子项时400px不够）
□ KPI卡片内不含 <td>/<tr>，标题/数值/标签全部用 <div>
□ 表格操作列（最后一列）有 white-space:nowrap
□ 超过4列的表格独占全宽，不放 grid2 两列并排
□ 菜单项全部有 data-page 属性，setPage() 用 dataset.page 精确匹配
□ 多文书类型（>6个）的 tab 容器设置 overflow-x:auto; white-space:nowrap
□ 无 menu-section 分区标签，改用 menu-divider 细分割线
□ 从技术要求逐一提取所有数据实体类型，在 JS 数据中全覆盖
```

### 7条约束详解

#### #1 KPI卡片禁用 `<td>/<tr>`

```html
<!-- ❌ 错误（会导致KPI卡片沉到页面底部） -->
<div class="kpi"><td class="kpi-lbl">待处理线索</td></div>

<!-- ✅ 正确 -->
<div class="kpi"><div class="kpi-lbl">待处理线索</div></div>
```
**原因**：`<td>` 放在 `<div>` 内，浏览器自动修复HTML语义时将 `<td>` 提升到外层，破坏 Flexbox/Grid 布局。

#### #2 操作列必须 `white-space:nowrap`

```css
.tbl td:last-child { white-space: nowrap; }
```
**原因**：多列表格每列宽度有限，"转线索"、"查看详情"等按钮文字会折成两行。

#### #3 侧边栏宽度 ≥ 248px

```css
.sidebar { width: 248px; }
```
**原因**：中文二级菜单项通常8-12个字，230px时容易被截断。

#### #4 sub-menu max-height ≥ 600px

```css
.sub-menu.open { max-height: 600px; }
```
**原因**：每个子菜单项约占40px高度，6个子项需240px+缓冲，400px时底部会被截断。

#### #5 菜单激活用 `data-page` 属性

```html
<div class="sub-item" data-page="risk-alert" onclick="setPage('risk-alert')">异常行为预警</div>
```
```javascript
function setPage(id) {
  document.querySelectorAll('[data-page]').forEach(el =>
    el.classList.toggle('active', el.dataset.page === id)
  );
}
```
**原因**：`onclick.toString()` 字符串格式因引号、空格、换行不稳定，高亮会偶发性失效。

#### #6 超过4列的表格独占全宽

```
列数 ≤ 4 → 可放 grid2 两列并排
列数 ≥ 5 → 必须独占全宽
```
**原因**：5列以上每列约100-120px，操作列严重拥挤。

#### #7 文书类型 > 6 个时 tab 容器横向滚动

```css
.doc-tabs {
  overflow-x: auto;
  white-space: nowrap;
  scrollbar-width: thin;
}
.doc-tab { display: inline-block; }
```
**原因**：14种文书类型放同一 tab 栏时，不设水平滚动会导致后面的 tab 溢出不可见。

**风险标记格式：**
```
⚠️风险标记：[违规类型] - [具体描述]

常见违规类型：
- 超量用药：药品剂量超出说明书推荐量
- 诊疗不符：费用与医疗行为不匹配
- 串换项目：以低价收费标准执行高价项目
- 挂床住院：患者实际不在院但按住院收费

- 重复用药：同类药品多渠道重叠取药
```

## 常见问题

**Q1：某些按钮点击没有反应？**
检查 JS 中函数是否正确导出（`window.funcName = funcName`），`onclick` 使用的函数名是否与 JS 定义一致。

**Q2：某个页面数据显示为空白？**
检查 `setPage()` 函数中是否有对应页面的渲染调用（如 `if(id==='xxx') renderXxx()`）。

**Q3：怎么判断哪些模块该保留、哪些该删除？**
核心问题："这个功能的使用者是谁？"
- 甲方用户日常操作 → 保留
- 乙方技术人员操作 → 删除
- 系统自动执行的过程 → 改为只读统计，删除操作按钮

**Q4：数据感觉不真实，像测试数据？**
要求 AI 使用：真实地名构造机构名、脱敏格式人名（王**）、真实规模数量级、近期日期、行业规范编码（CL-202603-0012、YB2024-000318）、真实医疗费用范围（门诊几百、住院几千到几万）。

**Q5：页面太多，一次生成质量不稳定？**
分模块生成，每次只生成2-3个页面，然后合并。合并时注意 CSS 变量和 JS 公共函数不要重复定义。
