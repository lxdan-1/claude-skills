# 提示词优化

## 使用目标

优化面向产品工作流的提示词，让模型不只是“回答问题”，而是稳定地产出可复用、可落地、能串联上下游的结果。

当前优先服务这些场景：

- `需求分析`
- `PRD写作`
- `交互页面生成`
- `评审检查`

## 适用场景

- 现有提示词输出不稳定、偏题或不够具体。
- 需要让模型按指定格式产出内容。
- 需要把模糊指令改写成清晰、可复用模板。
- 需要让多个 skill 串成一条工作流，而不是每步都重讲一遍要求。
- 需要让模型输出与目录约定、文件命名、配套交付方式保持一致。

## 优先优化的 5 个维度

1. 任务目标：模型到底要产出什么。
2. 上下文输入：应从哪些已有材料继续，而不是从零猜。
3. 约束要求：格式、目录、文件类型、命名、配套关系。
4. 输出结构：最终结果长什么样，是否可直接复用。
5. 验收标准：什么样算完成，什么样算不合格。

## 执行步骤

1. 提取原始目标：到底希望模型完成什么任务。
2. 判断所属工作流节点：需求分析、PRD、原型、评审，还是跨节点串联。
3. 识别缺口：缺少上下文、角色、约束、示例、目录规则还是输出格式。
4. 补足上下游衔接：说明该提示词应该接收什么输入，并产出什么供下一步使用。
5. 重写结构：目标、背景、输入、限制、输出格式、质量标准。
6. 保留必要自由度，避免限制过死导致回答僵硬。
7. 给出优化前后对比和适用说明。

## 工作流优化原则

### 需求分析类提示词

- 要强调先做类型判断
- 要强调补齐缺失信息和待确认项
- 输出要能直接给 PRD 继续使用

### PRD写作类提示词

- 要强调承接需求分析结果
- 要强调页面型需求与 HTML 原型的配套关系
- 要明确 PRD 输出目录和交付建议

### 交互页面生成类提示词

- 要强调默认直接生成可运行 HTML
- 要强调与 PRD 同目录放置
- 要强调真实合理模拟数据和可操作交互

### 评审检查类提示词

- 要强调问题分级
- 要强调 PRD 与 HTML 原型的一致性评审
- 要强调输出问题、影响、建议，而不是泛泛评价

## 输出要求

优先输出以下结构：

```markdown
## 原提示词问题

## 优化思路

## 优化后提示词

## 使用说明

## 可选增强项
```

## 推荐模板

```markdown
## 原提示词问题
- 问题1：
- 问题2：

## 优化思路
- 补充了哪些上下文：
- 增加了哪些约束：
- 放宽了哪些不必要限制：

## 优化后提示词
[可直接复制使用的完整提示词]

## 使用说明
- 适用场景：
- 需要补充的输入：
- 预期产出：

## 可选增强项
- 增强项1：
- 增强项2：
```

## 质量检查

- 是否明确任务目标和边界。
- 是否指定必要上下文，而不是堆背景材料。
- 是否定义输出格式和成功标准。
- 是否减少歧义词和相互冲突的要求。
- 是否保留可复用性，避免只适配单次问题。
- 是否明确了上下游衔接，而不是孤立完成当前步骤。
- 是否体现了目录、文件命名和配套交付约定。
