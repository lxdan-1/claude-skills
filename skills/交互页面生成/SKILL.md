# 交互页面生成

## 使用目标

把 `需求分析` 或 `PRD写作` 的结果，进一步转成可直接打开预览的 HTML 原型文件，让设计、前端和评审人员不仅能看结构，还能实际点击、输入、切换和验证交互。

## 输入前提

优先接收以下输入：

- 页面目标或功能目标
- 适用角色
- 核心场景
- 主流程
- 规则与约束
- 字段与数据依赖
- 范围界定

如果这些信息还不完整，优先回到 `需求分析` 或 `PRD写作` 补结构，再来生成页面方案。

## 适用场景

- 需要从需求出发生成页面结构和交互说明。
- 需要梳理表单、列表、详情、弹窗、画布等页面模块。
- 需要为原型、UI 或前端实现提供页面级说明。
- 需要把 PRD 中的功能描述转换成可落地页面方案。
- 需要直接产出 `.html` 文件供预览、演示或评审。

## 先做判断

开始前，先判断页面类型：

### 列表页

常见模块：

- 筛选区
- 列表区
- 批量操作区
- 分页区
- 页签区

重点关注：

- 查询条件
- 列表字段
- 行操作
- 批量操作
- 空态和禁用态

### 表单页

常见模块：

- 基础信息区
- 表单字段区
- 上传区
- 操作按钮区

重点关注：

- 字段类型
- 必填与校验
- 联动关系
- 保存与取消
- 成功/失败反馈

### 详情页

常见模块：

- 基础信息区
- 信息分组区
- 历史记录区
- 操作区

重点关注：

- 展示规则
- 只读/可编辑差异
- 状态变化后的可见性

### 弹窗或侧边抽屉

重点关注：

- 打开条件
- 尺寸和位置
- 提交/关闭逻辑
- 是否阻断主流程

如果一个需求同时包含多个页面，不要混写成一个大块。优先拆为“列表页”“新增/编辑页”“详情页”“弹窗”分别说明。

## 默认产物要求

除非用户明确只要文档说明，否则默认直接创建可运行的 HTML 文件，而不是只输出 Markdown 页面方案。

默认要求如下：

- 产物格式优先为单文件 `.html`
- 默认使用原生 HTML + CSS + JavaScript
- 样式、脚本、模拟数据优先内嵌，保证开箱即用
- 页面打开后无需额外依赖即可本地预览
- 页面中的核心按钮、筛选、切换、弹窗、表单校验等交互必须可操作

默认输出位置：

- 优先与对应 PRD 文件放在同一目录
- 仅在尚未形成明确 PRD 文件时，才输出到 `05-项目输出/临时输出/交互原型/`

目录使用原则：

- 如果已有对应 PRD 文件，HTML 原型默认与该 PRD 放在同一目录，形成成套交付
- 如果 PRD 位于 `05-项目输出/阶段产出/`，原型也默认放在同目录
- 如果 PRD 位于 `05-项目输出/临时输出/`，原型也默认放在同目录
- 如果一个需求包含多个页面，优先在 PRD 所在目录下按需求名称建立子目录或使用统一前缀命名
- 只有在当前阶段还没有明确 PRD 文件时，才退回到 `05-项目输出/临时输出/交互原型/`
- 生成文件时，应优先使用可读文件名，避免 `index1.html`、`test.html` 这类弱语义命名

如果需求较大，可拆成多个 HTML 文件，但必须说明入口文件和页面跳转关系。

## 执行步骤

1. 明确页面目标：这个页面帮助用户完成什么任务。
2. 识别用户路径：入口、主操作、反馈、退出分别是什么。
3. 判断页面类型，决定页面应拆成哪些模块。
4. 拆页面结构：顶部、左侧、中部、底部、弹层、抽屉等区域。
5. 补交互逻辑：按钮状态、联动关系、切换逻辑、跳转逻辑。
6. 补状态设计：默认态、加载态、空态、错误态、禁用态、成功态。
7. 标注关键规则：字段校验、按钮权限、操作限制、状态切换。
8. 设计真实合理的模拟数据，覆盖主路径和关键分支。
9. 直接输出 HTML 原型文件，必要时补简短说明。

## 页面拆解清单

生成页面方案时，优先覆盖这些维度：

- 页面入口
- 页面目标
- 适用角色
- 页面区域划分
- 模块说明
- 主路径交互
- 分支路径交互
- 字段与校验
- 按钮与权限
- 状态与反馈
- 异常与边界处理
- 模拟数据设计

## 和前置 skill 的衔接规则

当输入来自 `需求分析` 时，按以下方式展开：

- `需求目标` -> 页面目标
- `适用角色` -> 角色与权限说明
- `核心场景` -> 页面核心任务和关键操作
- `主流程` -> 页面主路径交互
- `规则与约束` -> 字段校验、按钮规则、联动关系
- `数据与依赖` -> 下拉来源、接口依赖、展示字段
- `范围界定` -> 页面本次覆盖范围与排除项

当输入来自 `PRD写作` 时，重点承接：

- `界面元素` -> 页面结构
- `业务用例` -> 页面模块与交互流程
- `业务规则` -> 交互限制、状态切换、联动规则
- `数据字典` -> 字段定义
- `权限说明` -> 按钮/页面可见性

## HTML 原型实现规则

生成 HTML 原型时，默认遵循以下原则：

### 技术实现

- 默认生成单个自包含 HTML 文件
- 使用语义化 HTML 结构
- CSS 与 JavaScript 直接写在文件内部
- 不默认引入 CDN、构建工具或第三方框架
- 只有在用户明确要求时，才改用 Vue、React 或外部依赖

### 交互要求

- 关键按钮必须可点击并有反馈
- 列表页至少支持筛选、重置、分页或页签切换中的主要交互
- 表单页至少支持输入、选择、校验、提交、取消中的主要交互
- 弹窗或抽屉必须能打开、关闭、提交和回填
- 状态切换必须真实可见，不要只做静态摆设
- 错误提示、成功提示、禁用态和空态至少覆盖主要场景

### 模拟数据要求

- 默认填充真实、具体、合理的数据，不使用“测试1”“示例A”“张三李四列表1”这类低质量占位
- 数据要符合业务语境，例如机构名称、日期、状态、金额、规则名称、人员姓名、编号等
- 至少准备 3 组以上列表数据，能支撑筛选、状态切换或详情展示
- 至少准备 1 组异常数据或边界数据，用于演示错误提示、空态或禁用态
- 表单默认值、下拉选项、详情信息、统计数字之间要相互一致

### 内容质量要求

- 中文页面默认使用符合业务语境的文案
- 不使用 lorem ipsum、foo/bar、mock data 这类占位内容
- 不只画壳子，必须让用户一眼看出主要业务动作
- 如果原始需求缺字段，先补合理假设，并在页面说明中标注“模拟设定”

### 文件交付要求

- 文件名应与页面用途对应，例如 `线索池页面原型.html`
- 如包含多个页面，命名应可读，例如 `线索池列表.html`、`线索新增抽屉.html`
- 如已存在对应 PRD 文件，默认与 PRD 文件同目录放置
- 如同一需求包含多个文件，建议与 PRD 放在同一目录下，并使用统一前缀或子目录管理
- 仅在没有对应 PRD 文件时，默认输出到 `05-项目输出/临时输出/交互原型/`
- 如需说明使用方式，只补充最短说明，不要重复写大段 Markdown 方案

## 输出要求

如果用户明确要文档说明，可使用以下结构：

```markdown
## 页面目标

## 页面类型

## 适用角色

## 页面入口

## 页面结构

## 模块说明

## 核心交互流程

## 状态与反馈

## 字段与规则

## 异常与边界处理
```

## 输出写法要求

- 围绕用户任务组织内容，不要堆功能点。
- 页面结构要有区域感，让人能想象布局。
- 交互流程优先描述主路径，再补关键分支。
- 字段说明尽量写清字段类型、是否必填、校验或联动。
- 按钮规则尽量写清“什么时候可点、什么时候禁用、点了会发生什么”。
- 如果原始信息不足，明确写待确认，不要擅自补 UI 细节。

如果是直接生成 HTML 文件，则优先把这些要求落实到代码和页面表现里，而不是只写成说明。

## 质量检查

- 是否围绕用户任务组织页面，而不是堆功能。
- 是否说明主路径和关键分支路径。
- 是否覆盖默认态、空态、错误态、权限受限等状态。
- 是否把页面元素和业务规则对应起来。
- 是否把按钮、字段、状态写成可实现表达。
- 是否能直接支持原型设计或前端实现。
- 如果输出的是 HTML，是否打开即可预览且关键交互真实可操作。
- 模拟数据是否具体可信，而不是模板占位。
